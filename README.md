# phr-personal-manager · 个人病历自主管理工具

基于 OMAHA 开放医疗与健康联盟《"当归"个人健康档案文件格式规范》v1.0 的**非官方第三方实现**。
拍下病历或检验报告，AI 帮你整理成结构化记录；病程按时间铺开，指标趋势一眼可见；
一键导出符合规范、可长期保存、可跨软件打开的 `.phr` 个人健康档案。

> **重要声明**
> 本工具仅对个人医疗资料做整理、呈现与溯源，**不提供诊断或治疗建议；健康问题请咨询医生**。
> 本项目与 OMAHA / 浙江数字医疗卫生技术研究院无隶属关系；「当归」为其项目名称，
> 规范原文见 [ChinaOMAHA/Angelica-PHR](https://github.com/ChinaOMAHA/Angelica-PHR)（MIT）。

## 特性

智能解析（图片病历 → 结构化记录，AI 产物强制人工确认后入档，全程记录算法溯源信息）；
横向病程时间轴（疾病泳道 × 时间轴，异常标红，点击展开）；疾病演进路径（阶段化事件链 +
现状整理 + 就诊沟通要点，阶段划分由内容模板驱动）；检验指标跨记录趋势图；`.phr` 标准导出
（容器结构、SHA-256 清单、官方 XSD 校验全部通过，见 CI）；数据全部保存在本机。

## 使用方式

**方式一 · claude.ai Artifact（零配置）**：将 `artifact/当归PHR管理工具.jsx` 作为 Artifact
在 claude.ai 中运行，存储与模型调用由平台代理，开箱即用。

**方式二 · 独立部署**：

```bash
npm install
npm run dev        # 开发
npm run build      # 构建，产物在 dist/
```

首次使用到「设置与导出 → 模型服务」填入 Anthropic API Key（可选自定义 Base URL 指向代理）。
数据存于浏览器 IndexedDB；**API Key 只保存在本机**。注意：上传的病历图片会发送到所配置的
模型服务用于解析；浏览器直连模型 API 仅适合个人自用，生产部署请改用自建服务端代理。

## 病友分享模块（默认关闭）

`src/config.js` 中的分享功能默认关闭。开启（构建时设 `VITE_ENABLE_SHARE=true`）意味着
部署者成为健康类用户生成内容的管理主体，须自行承担内容审核与所在地法规的合规责任。
项目内置发布前 AI 三级审核（放行/提示/拦截）作为辅助，不能替代部署者的审核义务；
独立部署下分享数据默认仅存本机，多人共享需自行接入同步后端。

## .phr 合规性

```bash
npm run validate:phr
```

用真实源码模块生成样例档案包并校验：mimetype 首条目不压缩、manifest SHA-256 逐项一致、
四个 XML 通过上游官方 XSD（随仓库附带于 `standard/schema/`，来源与许可见该目录说明）。
CI 在每次提交时自动执行构建与上述校验。

与规范的关系：容器封装、信封结构、算法溯源严格遵循规范；内容模板集与 `content.json`
结构化模型为本实现在规范留白处的自有扩展（codeSystem 暂为自声明 URN）；manifest 不含自身
哈希系对规范模糊处的解释（自引用哈希逻辑上不可实现）。完整符合性声明见
[docs/当归PHR-架构设计说明与评估.md](docs/当归PHR-架构设计说明与评估.md) 第 2 节。

## 架构

Vite + React 单页应用，四层：表现层（六个页面）→ 业务层（解析/时间轴/演进/分享）→
内核层（统一 record 数据模型 + 内容模板注册表，借鉴 openEHR 双层思想：信封稳定、模板演进）
→ 适配层（`src/adapters/`：存储 Artifact↔IndexedDB、模型调用 代理↔API Key）。
详见 [docs/当归PHR-MVP开发文档.md](docs/当归PHR-MVP开发文档.md) 与
[docs/当归PHR-架构图.svg](docs/当归PHR-架构图.svg)。

## Roadmap

后端化（分享区服务端审核与发布者自管）→ 术语编码（指标别名 → LOINC/ICD 子集）→
模板 codeSystem 正式 OID 注册 → `.phr` 导入回读 → 文档数字签名。
