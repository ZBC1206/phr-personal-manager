# 当归 PHR 工具 · 架构设计说明与评估

版本 v0.4 · 2026-07 · 配套文档：《当归PHR-MVP开发文档》（接口与数据字典细节以开发文档为准，本文侧重设计思路与优缺点评估）

---

## 1. 定位与总体架构

本工具是一个面向个人的病历自主管理应用，让用户把散落的纸质与图片病历汇聚为一份可携带、可溯源的个人健康档案。文件格式**基于** OMAHA 开放医疗与健康联盟《"当归"个人健康档案文件格式规范》v1.0（浙江数字医疗卫生技术研究院发起，开源于 ChinaOMAHA/Angelica-PHR）**调整实现**：容器结构、信封语义与溯源机制严格对齐官方 XSD，同时在规范留白处做了自有的实现层扩展（见第 2 节）。

MVP 形态为纯前端单页应用，自上而下分四层：**表现层**（档案 / 上传解析 / 病程时间轴 / 知识图谱 / 病友分享 / 设置导出六个页面）；**业务层**（M1 智能解析、M2 时间轴聚合、M3 图谱推导、M4 病友分享四个模块）；**内核层**（统一数据模型 record + 内容模板注册表，四个模块共用一份数据，不做二次转换）；**基础设施层**（本地持久化存储、共享存储区、.phr 组包器、视觉大模型 API）。用户病历数据全部留在本机，只有两类内容离开设备：解析时上送的单据图片（不附带身份字段），以及用户主动发布的分享内容（经知情确认与 AI 审核）。

## 2. 与 OMAHA 规范的关系（符合性声明）

| 层面 | 规范的立场 | 本工具的做法 | 性质 |
|---|---|---|---|
| 容器封装 | ZIP 容器，mimetype 首条目不压缩 | store-only ZIP，mimetype 为 `application/phr+zip` | 遵循 |
| 信封结构 | meta / subject / content / manifest 四个 XML，元素与顺序由 XSD 定义 | 逐一生成并以官方 XSD 校验，导出前不合规即失败 | 遵循 |
| 算法溯源 | AI 产物须标记 `generationMode=算法推导` 并附 processingInfo | 每条 AI 解析记录写入算法名、版本、处理时间、置信度 | 遵循 |
| 内容模板 | component 预留 `template` 元素（code/codeSystem/codeSystemName），模板内容由实现者定义 | 自定义七类内容模板集，codeSystem 暂用自声明 URN（`urn:angelica-phr:mvp-templates`） | 规范留白，自有扩展 |
| 内容文件格式 | 规范只管信封与引用，不约束内容文件内部结构 | `content.json` 采用自定义结构化模型（labs/dx/rx/sx/ex） | 规范留白，自有扩展 |
| manifest 范围 | 原文为"除 mimetype 与 signatures.xml 外"，未提及 manifest 自身 | 参照 ODF 惯例将 manifest 自身一并排除（自引用哈希在逻辑上不可实现） | 模糊处的解释选择 |
| 签名与加密 | signatures.xml / encryption.xml 为可选模块 | 不实现（依赖 CA 体系与密钥管理，超出个人工具边界） | 边界内不做 |

这张表同时是将来与其他 .phr 实现做互操作测试的对照基线；manifest 自引用问题建议整理后向上游仓库反馈。

## 3. 双层结构设计（借鉴 openEHR）

设计动机：早期版本中，"血常规里有哪些字段、处方单该抽什么"这类临床知识硬编码在三处代码里（抽取 prompt、核对表单、图谱推导），每支持一种新单据都要同步改三处。v0.3 借鉴 openEHR 参考模型与原型分离的思想，把系统切成两层：**信封层**（record 结构、容器封装、溯源机制）保持稳定，软件只面向这一层开发；**内容语义层**收敛为一份模板注册表——每个模板是纯数据，含身份（code/name/version）、抽取重点（promptHint，动态注入解析 prompt）、表单字段顺序（sections，驱动核对页动态渲染）、图谱实体门控（graph）。七类记录类型各配一个 v1.0 模板。

数据流是：AI 解析时把七个模板的抽取重点一次性注入 prompt，模型先判型再按模板重点抽取；核对页按判定类型的模板排布字段，用户切换类型即实时换模板；入档时把模板快照（code+version）固化到记录上；导出时写入规范预留的 `template` 元素。新增单据类型 = 在注册表里加一份数据，三处行为自动跟随，代码零改动。与 openEHR 生态互操作时，只需在模板层维护一张"本模板 ↔ openEHR 原型 / FHIR 资源"的映射表，无需引入 ADL/AQL 全栈。

## 4. M4 病友分享架构

分享区是全工具风险最高的模块，架构上用三道护栏收束：**来源分层**——每条内容强制声明"患者经验"或"指南·权威资料"（后者必填出处），经验类永远带"不构成医疗建议"标识；**发布前 AI 三级审核**——block（鼓动停药换偏方、指令性剂量方案、他人隐私、辱骂、广告导流：拦截并返回修改建议）、warn（个人治疗细节仅自述：放行加"谨慎参考"提示）、ok；**主权隔离**——病历数据永不随分享外泄，采纳的经验只存本地、不进 .phr 导出。存储上，分享条目走共享区（发布前强制知情勾选），"有用"与"举报"做成一键一票的独立共享键、由枚举聚合计数以规避并发覆盖，三人举报自动隐藏；不设评论区，把互动复杂度和审核成本压到最低。

## 5. 优势

**数据主权与隐私面。** 病历数据不出本机、不建账号、主体信息只收三项非敏感字段，这既是"当归"理念的直接落实，也让工具在没有任何合规资质的前提下可以安全地做业务验证——个人健康数据一旦上服务器，监管与安全成本会立刻上一个量级，纯前端形态把这个成本推迟到了真正需要的时候。

**可演进性面。** 双层结构让临床内容模型与代码解耦：模板按类型收窄抽取指令后，识别准确率通常好于"一把抓"的通用 prompt；模板快照固化在记录上，模板将来升级不会歪曲历史数据的语义；这套机制同时是通往 openEHR/FHIR 互操作的接口预留。

**可信面。** 三条溯源链贯穿全产品：AI 解析产物带 processingInfo（算法/版本/置信度）、图谱每条边可回指原始记录、导出文件带 SHA-256 清单并通过官方 XSD 校验。加上"模型输出永不直接入档"的强制人工确认环节，工具在"AI 可能出错"这一前提下仍然可用、可核查。

**工程面。** 单文件、零后端、零部署，改一处即全量发布；四个模块共用一份数据模型，没有同步与转换成本；已验证的组包逻辑与 UI 层物理分离，UI 迭代不触碰合规代码。

## 6. 局限与风险

**纯前端形态的天花板（最主要的一组）。** 共享存储只有最终一致与 last-write-wins 语义，投票计数在并发下可能少算；匿名标识存在本地、清空即换人，无法防止刷票或重复举报；AI 审核在客户端发起，技术上可被绕过（生产形态必须移到服务端）；发布者目前不能编辑或删除自己的分享（只能靠举报机制下架），这在上线真实用户前必须补上；本地存储有单键与总量上限，图片虽经压缩，数百条记录后仍会触顶。

**语义层的欠账。** 未做术语编码（ICD/LOINC），"血红蛋白"与"Hb"会在趋势图和图谱里分裂成两个实体——记录量越大这个问题越疼，是下一阶段优先级最高的数据质量工作；图谱推导规则刻意保守（只做记录内共现关系），不做跨记录的因果推断，换来可解释性但牺牲了发现能力。

**对标准的依赖与解释风险。** codeSystem 用自声明 URN 而非注册 OID，跨实现交换时模板身份可能撞名；manifest 自引用是我们的单方解释，若官方后续给出不同澄清需要跟进调整；不做签名意味着 .phr 文件的完整性校验只防传输损坏、不防恶意篡改。

**其他。** 解析与审核依赖在线模型调用，离线不可用，且置信度是模型自评、并非统计校准值；图谱力导向布局为 O(n²)，节点数百级后需换算法；单文件已约 1600 行，继续长需要拆分工程化；分享区冷启动依赖种子内容运营。

## 7. 演进路线（局限 → 对策）

| 局限 | 影响 | 对策与时机 |
|---|---|---|
| 共享区弱一致 / 审核可绕过 / 不能删自己的帖 | 社区可信度 | 生产形态后端化：kb 三表 + 服务端审核流水线 + 发布者自管（M1 阶段） |
| 术语未编码 | 趋势与图谱实体分裂 | 引入指标别名映射表起步，逐步对接 LOINC/ICD 子集（M2 阶段） |
| 本地存储上限 | 记录量天花板 | 附件分级压缩 + 导出归档提示；生产形态转对象存储 |
| codeSystem 无正式 OID | 跨实现模板撞名 | 生产化前向标准组织申请 OID；同步向 OMAHA 反馈 manifest 自引用问题 |
| 无签名 | 防篡改缺失 | 生产形态评估接入第三方 CA 的文档签名服务 |
| 单文件规模 | 维护性 | 后端化同期拆分为模块化工程 |

## 附：v0.4 变更记

本次交互重构后，第 1 节所述"M3 图谱推导"更替为"M3 疾病演进路径"：不再构建实体共现图谱，而是按疾病梳理阶段化事件链（阶段划分由模板层 stage 字段驱动），并在路径下自动生成现状整理与就诊沟通要点；M2 时间轴改为疾病泳道式横向布局，可视化优先、点击展开；M4 分享按疾病自动匹配进演进视图，采纳状态双向同步。第 5 节"可信面"的图谱溯源表述相应调整：演进路径上的每个节点即原始记录本身，溯源粒度不降反升。第 6 节实体图谱"发现能力有限"的局限随视图移除而关闭；术语未编码导致的指标分裂在演进视图中同样存在，优先级不变。
